## Key Takeaways

### What Makes a Good PR?

1. ✅ **Clear title and description** - Explains what and why
2. ✅ **Issue linkage** - Traces back to requirements
3. ✅ **Proper imports** - Uses shared packages
4. ✅ **Comprehensive tests** - Unit, E2E, visual
5. ✅ **Documentation** - Screenshots, updated docs
6. ✅ **Security** - No secrets, vulnerabilities fixed
7. ✅ **Code quality** - Linted, formatted, reviewed

### Common Pitfalls to Avoid

1. ❌ **No description** - Use the template
2. ❌ **Local imports** - Use @iiskills/ui instead
3. ❌ **No tests** - Always add tests
4. ❌ **Committing .env** - Use .env.example
5. ❌ **No screenshots** - Required for UI changes
6. ❌ **Large PRs** - Break into smaller chunks
7. ❌ **Hardcoded secrets** - Use environment variables
